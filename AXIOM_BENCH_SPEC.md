# AXIOM-Bench 0.1 Specification

Status: proposed normative benchmark contract  
Owning milestone: M1 / issue #12  
Benchmark series: `0.1.x`  
Product scope: AXIOM v0.7 semantic-oracle subset

## 1. Purpose

AXIOM-Bench 0.1 establishes a reproducible before-state for AI-driven
programming before AXIOM adds new language features. It tests benchmark
methodology and the existing v0.7 subset. It cannot prove that AXIOM is generally
better than Rust, Zig, or Go.

The benchmark answers five separate questions:

1. Can a model produce source that parses and compiles from a bounded language
   pack?
2. Can it satisfy public and acceptance behavior, including edge cases?
3. Does ordinary compiler feedback help it repair failures within three model
   iterations?
4. Does structured compiler feedback add value when that lane becomes available?
5. Can a full repository agent complete the same task under equivalent tool
   capabilities?

No single composite score may collapse these questions.

## 2. Normative artifacts

A frozen suite contains:

```text
suite.json
schemas/
language-packs/
tasks/
fixtures/
toolchains.json
SHA256SUMS
```

A run produces:

```text
run.json
attempts/<task>/<language>/<attempt>.json
traces/<task>/<language>/<attempt>.jsonl
raw-completions/<task>/<language>/<attempt>.txt
workspaces/<task>/<language>/<attempt>/patch.diff
commands/<task>/<language>/<attempt>/*.json
summary.json
SHA256SUMS
```

All JSON artifacts use UTF-8, sorted object keys when generated by AXIOM tools,
newline termination, explicit `document_kind`, and explicit `schema_version`.

## 3. Benchmark identity and immutability

The initial frozen version is:

```text
0.1.0
```

A suite identity consists of:

- benchmark semantic version;
- canonical suite JSON SHA-256;
- SHA-256 for every task, language pack, fixture, schema, and toolchain record;
- repository commit;
- creation timestamp;
- freeze timestamp;
- provenance statement.

After freeze:

- task prompts cannot be edited;
- public or acceptance tests cannot be weakened or replaced;
- language packs cannot be made task-specific;
- reference and seeded-wrong solutions cannot change;
- scoring rules cannot change;
- a correction requires a new benchmark version;
- previous versions and all negative results remain available.

Patch releases may correct runner defects only when they do not alter candidate
acceptance. Any change that can change task success requires a minor version.

## 4. Result lanes

### 4.1 Language-only

Purpose: isolate source-language and ordinary-documentation predictability.

Permitted context:

- task prompt;
- task files visible to every language;
- the frozen equal-spec language pack;
- ordinary text compiler diagnostics;
- ordinary public-test output;
- at most three model responses.

Forbidden:

- AXIOM-only semantic explain documents;
- hidden test contents;
- task-specific solution fragments;
- unequal repository retrieval;
- language-specific human repair advice;
- automatic source rewriting unavailable to every language.

### 4.2 Compiler-assisted

Purpose: measure the effect of standard machine-readable compiler output.

Every language receives only structured outputs available through its frozen
ordinary toolchain configuration. The run records exactly which documents were
returned and their byte/token cost.

M1 may validate this lane's schema even when AXIOM v0.7 has not yet delivered the
complete M3 protocol. Missing structured capability is represented explicitly,
not filled by custom human hints.

### 4.3 Full-agent

Purpose: measure repository-development performance.

Equivalent tools:

- list files;
- read bounded files;
- search bounded text;
- create/edit/delete files;
- format;
- build/check/test;
- read frozen compiler output;
- inspect bounded history when the task permits it.

The agent cannot access acceptance tests, reference solutions, seeded-wrong
solutions, benchmark control files, secrets, network credentials, or other
language variants.

### 4.4 Runtime and compiler observations

Runtime and compiler measurements are separate observational lanes. They never
compensate for failed functional or security acceptance.

## 5. Iteration contract

A benchmark task has exactly three model iterations unless a stricter task
budget is declared.

One iteration is:

1. assemble permitted context;
2. obtain one model response;
3. retain the raw response unchanged;
4. extract declared edits without silently repairing them;
5. apply edits in a fresh or previous-attempt workspace according to the lane;
6. run formatter/check/build/public tests under frozen commands;
7. capture every command, exit code, bounded stdout/stderr, duration, and file
   mutation;
8. return only permitted bounded feedback to the next model response.

Automatic formatting does not consume a model iteration. Any new model response
after feedback does.

Success after attempt one stops further model calls but still executes the full
acceptance and security checks. Failure after attempt three is final.

## 6. Task contract

Every task declares:

- globally unique task ID;
- task family;
- task version;
- supported language variants;
- prompt path and hash;
- visible starter files;
- candidate-edit allowlist;
- public/base checks;
- acceptance checks;
- security checks where applicable;
- resource budgets;
- expected entry point or project command;
- reference solution for harness self-test;
- at least one plausible seeded-wrong solution;
- contamination and provenance metadata;
- equivalence review record;
- explicit non-goals.

A task is valid only when:

- every language variant represents the same observable contract;
- the reference solution passes all checks;
- every seeded-wrong solution fails at least one acceptance/security check;
- public checks alone do not accept every seeded-wrong solution;
- task setup is unattended and deterministic;
- no language variant requires an undeclared remote dependency;
- toolchain commands are argument arrays, not shell strings;
- all relative paths remain within the task workspace.

## 7. Public versus acceptance checks

Public checks help the model understand the interface and ordinary examples.
Acceptance checks evaluate withheld edge cases inside the runner environment.

M1's seed acceptance tests are **execution-hidden**, not secret holdouts:

- they are absent from the candidate workspace and model context;
- they may exist in the public benchmark repository after freeze;
- contamination metadata therefore marks them `public_after_freeze`;
- they cannot support the final M13 superiority claim.

M13 requires a separately controlled holdout process.

## 8. Language variants and equivalence

Primary AI-development languages:

```text
axiom
rust
zig
go
```

Each variant declares:

- exact toolchain identity;
- source file convention;
- formatter command;
- check/build command;
- public-test command;
- acceptance-test command;
- expected diagnostic/exit behavior;
- language-pack ID;
- variant-specific unavoidable differences.

Equivalence is behavioral, not textual. Variants may use idiomatic declarations
where those do not change task difficulty or expose extra solution guidance.

A task-equivalence review must reject variants that differ materially in:

- required algorithm;
- visible edge cases;
- state or mutation requirements;
- error behavior;
- available standard APIs;
- hidden acceptance cases;
- amount of starter implementation;
- compiler/test feedback quality not attributable to the language/toolchain.

## 9. Equal-spec language packs

A language pack contains only concepts required by its task family. It declares:

- language and exact compiler version;
- standard-library/toolchain version;
- covered syntax and semantics;
- relevant APIs;
- deterministic examples by concept;
- safety and unspecified-behavior warnings;
- UTF-8 byte count;
- Unicode word count;
- line count;
- provider-token counts when a live model adapter supports exact tokenization;
- source hashes and reviewer decision.

Rules:

- no task identifiers, constants, algorithms, or hidden edge cases;
- concept-equivalent examples rather than translated solutions;
- equal preregistered ceiling for each live model/tokenizer;
- unused budget is not filled with irrelevant material;
- packs freeze before live seed execution;
- natural-knowledge runs remain separately labeled and cannot replace equal-spec
  results.

## 10. Raw output and extraction

The raw model completion is immutable evidence.

If an adapter extracts source or patches:

- extraction rules are frozen and language-neutral where possible;
- the extracted artifact is stored separately;
- extraction cannot synthesize missing syntax, imports, types, or error handling;
- rejected/ambiguous extraction is a recorded failure;
- raw completion hashes remain part of the attempt result.

A sanitizer that makes a candidate more correct is prohibited.

## 11. Command execution and trust boundary

Trust classes:

```text
trusted_reference
trusted_seeded_wrong
replay_only
untrusted_model_output
```

The built-in local executor is not a security sandbox.

It may execute only repository-controlled reference/seeded-wrong fixtures in a
disposable CI job or an explicitly trusted checkout. It enforces timeout,
working-directory, output-size, environment, and path limits as reliability
controls only.

`untrusted_model_output` requires a configured isolated sandbox backend whose
identity and policy are recorded. Without one the runner fails before applying
or executing candidate code with:

```text
AX-BENCH-SANDBOX-REQUIRED
```

A later Inspect AI adapter may use Docker or another approved sandbox extension.
Inspect's `local` environment is never accepted as isolation.

## 12. Resource budgets

Every task/lane freezes:

- maximum model iterations: 3;
- maximum model input/output tokens per iteration;
- maximum cumulative tokens;
- maximum tool calls;
- maximum compiler/test invocations;
- command timeout;
- total task wall clock;
- stdout/stderr byte cap;
- maximum candidate files;
- maximum candidate bytes;
- maximum changed lines;
- maximum retained feedback bytes.

Exhausting a budget is a task failure with a stable reason. The runner does not
continue off-record.

## 13. Trace contract

Every meaningful event is append-only JSON Lines with a monotonically increasing
sequence number:

- run/task/attempt start and finish;
- context assembly;
- model request and response metadata;
- raw completion stored;
- extraction result;
- file snapshot and mutation;
- command start and finish;
- public/acceptance/security check result;
- budget update/exhaustion;
- error, timeout, crash, cancellation;
- score decision.

Trace events include hashes for referenced payloads rather than duplicating
large content. The referenced payload must exist in the result bundle.

Timestamps are retained in raw traces but excluded from canonical semantic
hashes. Replay validates event order, hashes, and final score without model
access.

## 14. Scoring

Primary task outcome:

```text
task_success_within_3_iterations
```

A success requires all of:

- candidate extraction/application succeeded;
- project builds from the recorded clean task state;
- public functional checks pass;
- acceptance functional checks pass;
- security checks pass;
- no forbidden dependency, capability, file, or tool was used;
- output and exit behavior match;
- all budgets remain within limits;
- trace and result bundle are complete.

Required attempt-level outcomes:

```text
parse_success
compile_success
public_test_success
acceptance_test_success
security_success
full_success
```

Required failure reasons include:

```text
invalid_completion
invalid_patch
forbidden_file
forbidden_dependency
compile_failure
public_test_failure
acceptance_test_failure
security_failure
timeout
resource_limit
sandbox_required
runner_error
```

Metrics remain per lane and per language. Missing data is not zero and must be
reported explicitly.

## 15. Contamination and provenance

Every task records:

- authored date;
- first public date;
- source origin;
- whether derived from a known benchmark;
- source repository/issue hashes when applicable;
- whether task text or acceptance tests were publicly visible before the tested
  model's cutoff;
- transformation history;
- reviewer identities or automated review evidence;
- seed/holdout classification.

M1 seed tasks are methodology fixtures. They are not treated as uncontaminated
merely because AXIOM is uncommon.

## 16. Reproducibility

A run is reproducible only when it records:

- suite/version/hash;
- exact repository commit;
- OS, architecture, CPU metadata where available;
- compiler/toolchain versions and installation artifact hashes;
- model provider/model/version;
- exposed generation settings;
- adapter version;
- sandbox type/configuration;
- environment allowlist;
- task order and random seed;
- all prompts, completions, patches, commands, outputs, and failures;
- token accounting source;
- complete result bundle hash.

Reference, seeded-wrong, and replay conformance runs must be deterministic apart
from explicitly excluded timing/timestamp fields.

## 17. M1 conformance adapters

M1 requires three provider-free adapters before live model integration:

### Reference adapter

Applies the frozen canonical solution. Every language variant must pass.

### Seeded-wrong adapter

Applies each plausible wrong solution. Every one must fail acceptance or security
checks for the intended reason.

### Replay adapter

Consumes a frozen attempt/trace bundle, verifies all referenced hashes and event
order, and reproduces the final score without contacting a model.

These adapters prove the benchmark, not model quality.

## 18. Framework adapters

Framework integrations are optional adapters over the canonical schemas.

An Inspect AI adapter must:

- map task/sample metadata without loss;
- preserve raw AXIOM trace/result artifacts;
- use an approved sandbox for untrusted code;
- avoid Inspect-only scoring semantics;
- record Inspect version and EvalLog hash;
- pass reference/seeded-wrong/replay conformance.

No framework adapter may redefine task success.

## 19. Release conditions for AXIOM-Bench 0.1.0

The suite freezes only when:

- all schemas validate themselves and every artifact;
- task IDs and hashes are unique;
- all language packs pass leakage/equivalence checks;
- reference adapters pass every language variant;
- every seeded-wrong fixture is rejected;
- clean-checkout unattended execution passes;
- repeated conformance runs reproduce canonical hashes;
- lane separation is enforced by schema and runner;
- untrusted local execution is blocked;
- raw failures are retained;
- source/toolchain provenance is complete;
- Agent B passes adversarial benchmark checks;
- the ordinary AXIOM repository proof remains green.

## 20. Prohibited claims

AXIOM-Bench 0.1 results may describe:

- harness validity;
- current v0.7 baseline behavior;
- model/task observations with full limitations;
- differences between plain and available compiler feedback.

They may not claim:

- universal language superiority;
- a statistically established AI-first advantage;
- security of local untrusted execution;
- contamination-free evaluation;
- v1 readiness;
- M13 holdout success.
